# 流水线状态机定义

> 本文档定义发帖流水线 6 个步骤的状态流转、触发条件、异常处理规则。  
> 技术实现时以此为准。

---

## 一、整体状态机

```
                    ┌─────────────────────────────────────────────────────────────┐
                    │                     一个发帖任务的生命周期                      │
                    └─────────────────────────────────────────────────────────────┘

 ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
 │  Step 1   │───▶│  Step 2   │───▶│  Step 3   │───▶│  Step 4   │───▶│  Step 5   │───▶│  Step 6   │
 │ 素材&平台 │    │ 文案生成  │    │ 图片重命名│    │ 封面制作  │    │ 水印处理  │    │ 上传&发布 │
 └──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
  用户操作         用户确认▲       自动推进         用户确认▲       自动处理        用户触发▲
                  (阻塞等待)                      (阻塞等待)      (异步并行)      (逐平台触发)
```

---

## 二、每步状态定义

### 通用状态枚举

每个 Step 都有以下几种状态：

| 状态 | 英文 | 说明 |
|------|------|------|
| 等待 | `pending` | 还没轮到这一步 |
| 执行中 | `running` | 系统正在自动处理 |
| 待确认 | `awaiting_confirm` | 需要用户操作（确认/选择/编辑） |
| 已完成 | `done` | 这一步完成 |
| 失败 | `failed` | 这一步出错 |
| 跳过 | `skipped` | 用户选择跳过（仅部分步骤支持） |

## 三、各步骤详细定义

### Step 1：素材 & 平台选择

```yaml
输入:
  - folder_path: string          # 素材文件夹路径
  - target_platforms: string[]   # 目标平台 ID 列表（至少 1 个）

自动行为:
  - 扫描 folder_path，统计文件：
    - images: 匹配 *.jpg, *.jpeg, *.png, *.webp
    - videos: 匹配 *.mp4, *.mov, *.avi, *.mkv
    - txt: 匹配 *.txt（文案文件）
  - 返回文件清单给前端展示

推进条件:
  - folder_path 非空 且 至少包含 1 个图片或视频
  - target_platforms.length >= 1
  - 用户点击「下一步」

输出（传给后续步骤）:
  - folder_path
  - target_platforms
  - file_manifest: { images: [], videos: [], txts: [] }
```

### Step 2：文案生成

```yaml
输入:
  - protagonist: string       # 主角
  - event: string             # 事件
  - photos: string            # 生活照描述
  - video_desc: string        # 视频内容描述
  - style: enum               # 文风（反转打脸风/搞笑玩梗风/麻辣吐槽风/深情共情风/悬念揭秘风）
  - author: string            # 作者
  - category: string[]        # 分类（多选）— 从目标平台的分类库动态加载，平台无分类库则为空

前置联动（来自 Step 1）:
  - 根据 target_platforms 读取各平台的分类库
  - 有分类库的平台 → 取并集展示分类选项
  - 所有目标平台都无分类库 → 隐藏分类字段
  - title_min_len: int        # 标题最少字数（默认 25）
  - title_max_len: int        # 标题最多字数（默认 30）
  - keyword_count: int        # 关键词个数（默认 10）
  - body_len: int             # 正文字数（默认 300）
  - paragraphs: int           # 段落数（默认 3）

自动行为:
  - 调用 copywrite_gen.py 生成文案
  - 返回：title, keywords, body

状态流转:
  pending → running（调 AI 生成） → awaiting_confirm（展示结果等用户确认）

用户操作:
  - 可编辑 title / keywords / body
  - 「重新生成」→ 重新回到 running
  - 「确认文案」→ done，自动推进 Step 3

输出:
  - confirmed_title: string
  - confirmed_keywords: string
  - confirmed_body: string
  - confirmed_author: string
  - confirmed_category: string[]
```

### Step 3：图片重命名

```yaml
输入:
  - folder_path: 来自 Step 1
  - prefix: string            # 默认值 = Step 2 的 confirmed_title（截取前 N 字符）
  - start_num: int            # 起始编号（默认 1）
  - digits: int               # 编号位数（默认 2）
  - separator: string         # 分隔符（默认 "_"）

自动行为:
  - 前端实时预览：旧文件名 → 新文件名
  - 用户确认后调用 image_rename.py 执行重命名

状态流转:
  pending → awaiting_confirm（展示预览） → running（执行重命名） → done

推进规则:
  - 用户确认 → 执行重命名 → 成功后自动进入 Step 4
  - 如果文件夹无图片 → 自动跳过（skipped）

输出:
  - renamed_files: { old: string, new: string }[]
```

### Step 4：封面制作

```yaml
输入:
  - folder_path: 来自 Step 1
  - layout: enum              # single / double / triple
  - size: string              # 封面尺寸 "1300x640" 等
  - head_margin: int          # 头顶留白百分比（默认 15）
  - quality: int              # JPEG 质量（默认 95）

自动行为:
  - 调用 make_cover.py 生成 2~3 个候选封面
  - 前端展示候选封面缩略图

状态流转:
  pending → running（生成候选） → awaiting_confirm（用户选封面）

用户操作:
  - 从候选中选一个
  - 「重新生成」→ 回到 running
  - 「手动选图」→ 打开文件选择器，用户选图后重新生成
  - 「确认封面」→ done

输出:
  - cover_path: string        # 最终封面文件路径
```

### Step 5：水印处理

```yaml
输入:
  - folder_path: 来自 Step 1
  - target_platforms: 来自 Step 1
  - cover_path: 来自 Step 4
  # 各平台的水印参数从「业务线配置」自动读取

自动行为:
  1. 读取每个目标平台的水印配置
  2. 前端展示各平台的水印方案卡片（供用户确认）
  3. 用户确认后，为每个平台并行处理：
     a. 图片水印：调用 image_watermark.py（N 个平台 = N 次调用，输出到各自子目录）
     b. 视频水印：调用 video_process.py watermark（N 个平台 = N 次调用）
     c. 封面水印：对封面也加上对应平台水印

状态流转:
  pending → awaiting_confirm（展示水印方案） → running（并行处理） → done

并行策略:
  - 各平台的水印处理互不依赖，可并行
  - 单个平台失败不阻塞其他平台
  - 前端实时显示各平台的处理进度

输出（按平台）:
  platform_outputs:
    - platform_id: string
      watermarked_images_dir: string
      watermarked_video_path: string
      watermarked_cover_path: string
      status: done | failed
      error: string | null
```

### Step 6：上传 & 发布

```yaml
输入:
  - 各平台的水印处理结果（来自 Step 5）
  - confirmed_title / keywords / body / author / category（来自 Step 2）
  # 注意：category 发布时按平台过滤，只传该平台分类库内的值；平台无分类库则不传
  - 各平台的发布 API 配置（从业务线配置读取）
  - 各平台的排版模板（从业务线配置读取）

每个平台独立执行以下子流程:
  1. 登录：调用 publish_api.py login（如已缓存有效 session 则跳过）
  2. 上传视频：调用 publish_api.py upload-video → 获得 video_id
  3. 等待切片：轮询视频切片状态，直到 100%
  4. 上传图片：调用 publish_api.py 上传带水印图片
  5. 组装帖子：按排版模板组装 HTML 内容
  6. 发布/存草稿：调用 publish_api.py publish

各平台子任务状态:
  - uploading（上传视频中，显示进度 %）
  - transcoding（切片中，显示进度 %）
  - ready（切片完成，等待用户触发发布）
  - publishing（发布中）
  - published（发布成功）
  - failed（失败 + 错误信息）

用户操作:
  - 单平台：切片完成后点「存草稿」
  - 批量：「一键发布所有已就绪平台」
  - 失败重试：点「重试」

状态流转:
  pending → running（并行上传） → awaiting_confirm（部分/全部切片完成，等用户触发发布）→ done
```

## 四、任务级状态汇总规则

任务的整体状态由各 Step 状态和各平台子任务状态汇总得出：

```
if 所有平台子任务 published        → 任务状态 = "已完成"
if 部分 published + 部分 failed    → 任务状态 = "部分完成"
if 所有平台子任务 failed           → 任务状态 = "失败"
if 任何 Step 在 awaiting_confirm   → 任务状态 = "待确认"
if 任何平台子任务在 transcoding    → 任务状态 = "切片中"
if 任何 Step 在 running            → 任务状态 = "进行中"
```

## 五、数据模型参考

```
Task（任务）
├── id: string
├── title: string（来自 Step 2 确认的标题）
├── folder_path: string
├── created_by: string（编辑用户 ID）
├── created_at: datetime
├── current_step: int（0~5）
├── status: enum
├── step_states: Step[]
│   ├── step: int
│   ├── status: enum
│   ├── started_at: datetime
│   ├── finished_at: datetime
│   └── data: json（各步骤的输入输出数据）
└── platform_tasks: PlatformTask[]（Step 5/6 按平台拆分）
    ├── platform_id: string
    ├── watermark_status: enum
    ├── upload_status: enum
    ├── upload_progress: float
    ├── transcode_status: enum
    ├── transcode_progress: float
    ├── publish_status: enum
    └── error: string | null
```
